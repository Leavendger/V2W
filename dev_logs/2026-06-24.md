## 今日完成

### Worker 健壮性（删除转写中文件的 bug 修复链）

用户反馈「删除转写中文件后重新上传一直排队」，多轮排查修复（每版都基于完整 Traceback 定位真因）：

1. **移除 `_cancelled` 取消机制**：原用 `_cancelled[file_id]` 标记取消，但 SQLite rowid 删除后会**复用**，新文件拿到同 id 被旧标记误杀（卡 uploaded）。
2. **transcribe 改可中断**：faster-whisper 段是生成器，新增 `transcribe_iter()`，worker 逐段消费 + 每段检查文件是否已删，删除则 `break` 中断 Whisper 推理（不再等整段跑完）。
3. **files.id AUTOINCREMENT**：SQLite 默认 rowid 删除复用 → 改 AUTOINCREMENT（启动迁移重建 files 表，保留数据），id 不再复用。
4. **删除检测改 SQL 直查**：worker 开头 `File.query.get` 把 file 对象**缓存进 session**，删除后 session 仍返回缓存 → `_is_file_deleted()` 改用原始 SQL 直查 `files` 表绕过 ORM 缓存。
5. **except 块稳健化**（致命根因）：删除时 worker 在 decode_audio → `FileNotFoundError` → except 设 failed + commit，但 file_record 已删 → commit 抛 `StaleDataError`（在 except 内未捕获）→ **worker 线程崩溃** → 队列永久卡死。改为 except 内先 SQL 查文件是否还在 DB，不在则跳过状态更新，保证 worker 存活。

### diarize 前 转 wav（说话人识别关键修复）

- 现象：上传「识别说话人」，卡片显示区分，但转写内容无 speaker（speaker 全 None）。
- 根因（Traceback）：mp3 文件含**损坏 MPEG 头**，pyannote 的 mpg123 解码失败（`Illegal Audio-MPEG-Header` → `Unspecified internal error`），优雅降级 speaker 留空。faster-whisper 用 ffmpeg 解码（容错）所以转写正常。
- 修复：diarize 前用 ffmpeg 转 16k mono wav 再喂给 pyannote。
- 验证：损坏 mp3 转 wav 后成功识别 **4 个说话人**（304 turns，29 分钟）。

### UX 优化

- **删除转写中文件的「终止」toast**：右下角弹 `正在终止之前的转写任务…` + 流动进度条，轮询 `/api/worker/status`，worker 不再处理该文件时显示 `之前的转写任务已关闭` 保留 3 秒淡出。解决大音频逐段中断有延迟时用户的「不知后台关没关」焦虑。
- **上传模式 segmented control**：勾选框改为「👤 识别说话人 / ⚡ 快速转写」互斥切换按钮，默认识别说话人，各自 hover 说明。
- **卡片说话人标识**：diarize 文件卡片显示紫色「👤 说话人」标签。
- **hidden flag 默认 value=1**：上传 diarize flag HTML 默认 1（不依赖 JS 初始同步），确保识别说话人默认生效。
