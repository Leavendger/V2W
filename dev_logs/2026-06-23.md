### 关键实现要点

1. **搜索逻辑收拢在 `doSearch()`**：防抖 300ms → fetch API → 命中段加 `.search-hit`、`↑/↓`+回车循环导航、`Esc`/`✕` 清除、计数 `当前/总数`
2. **高亮由前端完成**（`highlight()` + `esc()` 转义）：规避 Python(Unicode 码点) 与 JS(UTF-16 码元) 在 emoji 文本上的索引错位
3. **与播放器共存**：新增 `searchActive` 标志，搜索激活期间 `timeupdate` 的 `scrollIntoView` 被守卫跳过（仍保留 `.active` 高亮），避免搜索定位与播放跟随互相抢占滚动
4. **P7 深链预留**：`detail.html` 读取 URL `?q=` 自动填入并触发搜索，P7 结果页可直接深链定位

## 关键决策记录

1. **LIKE 而非 FTS5**：单用户、单文件数百段量级，LIKE 毫秒级且中文子串匹配天然友好；FTS5 需纠结中文 tokenizer，得不偿失，留作升级路径
2. **高亮前端化**：后端只返回命中段落、不返回 `match_offsets`，消除跨语言索引差异风险

### 环境踩坑历程（重要，记入 design 第 9 节）

py3.13 下说话人分离两版均卡：

1. **pyannote 4.x**（清华源装的最新）：引入 `speaker-diarization-community-1` 子模型，需 **request access**（非即时同意，不可靠）。
2. **pyannote 3.1.1**（降级）：py3.13 下与 numpy 2（np.NaN 移除）/ torchaudio 2（set/get_audio_backend + torchaudio.backend 模块重构）/ 新 huggingface_hub（use_auth_token 移除）/ setuptools 80+（pkg_resources 移除）共 5+ 处不兼容，patch 无底洞。

**最终方案**：降级 **Python 3.11**（conda create -p venv python=3.11，机器已有 miniforge）+ **黄金依赖组合**（torch 2.0.1 / torchaudio 2.0.2 / numpy<2 / hf_hub<0.20 / setuptools<70）。pipeline 加载 90s，diarize 1 分钟音频 48s。

**网络**：pip 走代理（7890）下 torch 卡 43min 无效 → 改清华源 + `env -u *_PROXY` 绕代理直连；模型权重 gated，走官方源 huggingface.co + 代理（hf-mirror 只能下 metadata）。
